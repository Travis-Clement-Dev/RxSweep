## Findings, ranked by severity

### F1 · Moderate · The memo violates its own writing register, and the P3-5 sweep missed it

The rendered memo contains four em dashes, three of them in prose the register rule squarely covers. D5 and the style guide both state that generated artifacts use no em dashes, and `plans/2026-07-09-phase3-ux.md` marks the P3-5 register sweep ("memo, export headers, UI strings") as complete.

Evidence: `grep -c "—" runs/20260710T055516Z/report.html` prints 4. The sources are the title "Memorandum — Formulary Sweep Findings" (`src/rxsweep/templates/report.html.j2:70`), the RE: line (`:75`), "Exhibit A — Findings register" (`:115`), and the null-NDC placeholder `{{ f.item_ndc or "—" }}` (`:129`, `:136`).

Root cause: the register rule was encoded in the AI prompts (`triage.py:171`, `chat.py:32`), which work, but the template's static copy was never swept against the same rule. The AI holds the standard; the template does not. `findings.md`, `findings.csv`, and `findings.xlsx` are all clean, so the gap is specific to the memo.

### F2 · Moderate · Citation jump scrolls to the row but the promised flash never fires

The style guide (section 6) promises "row-flash on citation jump" and PR #1 claims citations "highlight/scroll to the finding row". Live, the scroll works and the highlight does not.

Evidence: clicking citation buttons in the chat brief scrolled `#finding-1` into the viewport, but polling `tbody tr.flash` every 50 ms for 1200 ms after the click never observed the class, across repeated attempts with both synthetic and real clicks.

Likely root cause: React Aria's dynamic collection caching. `FindingsTable.tsx:72` renders rows through `TableBody items={sorted}` with a render function whose closure captures `flashRow`, and React Aria only re-renders cached rows when the item or a declared dependency changes. The component never passes the documented `dependencies={[flashRow]}` prop, so the class update is invisible to the collection. Phase 3's hand-rolled table re-rendered on every state change, which is why this reads as a regression introduced by the port (the phase-3 branch was not re-verified live). No fix was applied per the brief.

### F3 · Moderate · The governance pack contradicts itself on how many jobs the AI has

SYSTEM_CARD.md says Claude has three jobs (adjudicate, summarize, chat) and is correct. GOVERNANCE.md's NIST GOVERN row still says "two jobs, nothing else" (`docs/GOVERNANCE.md:22`), and DESIGN.md says both: "three bounded jobs" in the Architecture section (`docs/DESIGN.md:36`) and "exactly two jobs. Nothing else." in the AI usage boundary section (`docs/DESIGN.md:91`). The Day 2 plan's wrap task updated the system card when chat shipped but not the other two documents. For a governance-first project the boundary statement is the single most load-bearing sentence in the pack, and it currently depends on which document you open.

### F4 · Moderate · The audit-log schema table is missing event kinds that real logs contain

GOVERNANCE.md documents eleven `kind` values. Real logs contain a twelfth, and the code can emit a thirteenth. Every post-phase-3 run writes an `exports` event (`pipeline.py:167`; present in 5 of 13 logs on disk), and `openfda.py:80` emits `fda_truncated` when a paged fetch hits the skip cap, which is exactly the disclosure README.md promises ("RxSweep discloses in the audit log if a fetch ever truncates"). Neither kind appears in the schema table, and the `ingest` event also carries an undocumented `columns` field. The compliance-file promise is that the schema table describes the log; two undocumented kinds is drift in the artifact that exists to prevent drift.

### F5 · Low · PR #2's bundle-growth figure does not reproduce

The PR comment says the bundle "grew ~130 KB minified with react-aria-components". Measured from the committed artifacts, the JS bundle went from 245,552 bytes on phase-3 to 544,460 bytes on this branch, which is roughly +299 KB minified, or +76 KB gzipped (77,134 to 152,717 bytes). Neither measure matches the stated figure. The brief lists the growth itself as a known accepted item, so the finding is only that the acceptance decision was recorded against a number roughly 2.3x smaller than the minified reality. The qualitative call (acceptable for a local tool) likely survives the corrected number, but that is Travis's call to re-make.

### F6 · Low · The openFDA disclaimer is verbatim in only one of the three required places

The live openFDA `meta.disclaimer` (fetched this session) has three sentences. DATA_PROVENANCE.md quotes it character-exact. The memo footer (`report.html.j2:186`) carries sentences one and two verbatim but omits the third ("We may limit or otherwise restrict your access..."), and the app shell (`web/src/App.tsx:52`) carries only the eight-word fragment "assume all results are unvalidated." DATA_PROVENANCE's own claim that RxSweep "displays it verbatim in every report" is therefore inaccurate as stated, and the brief's requirement of verbatim text in the app shell fails. Dropping the access-restriction sentence from user-facing surfaces is a defensible edit, but then the claim should say "displayed in relevant part," not "verbatim."

### F7 · Low · The brief's secret check cannot pass as written; the substantive check passes cleanly

`git log -p | grep -c "sk-ant"` prints 6, not 0. All six occurrences are documentation placeholders: the README's `ANTHROPIC_API_KEY=sk-ant-...` instruction (present since Day 1), and the audit brief quoting its own check. A strict scan for real key material, `git log -p --all | grep -cE "sk-ant-[A-Za-z0-9_-]{20,}"`, returns 0, and `.env` was never tracked on any branch. Conclusion: no secret ever entered history, and the literal check in the brief needs rewording because any repo that documents the key format will trip it.

### F8 · Low · The rubric's provenance is stated two different ways

README.md, DESIGN.md, and the app and memo footers say the severity rubric is "human-authored" or that "a pharmacist authored it." SYSTEM_CARD.md, JOURNAL.md, and the docstring at `triage.py:199` say it was "reviewed and approved by Travis Clement, PharmD," and the Day 1 plan shows the checkpoint where the table was scaffolded for him to author. Authored and approved are different provenance claims, and in an AIGP governance portfolio the difference is precisely the kind an examiner probes. One formulation should win everywhere; "reviewed and approved" appears to be the accurate one.

### F9 · Low · Chat requests log the question, not the full prompt

`chat.py:62` logs `prompt=question` for chat `ai_request` events, while the model actually receives the system prompt, the findings context, and the conversation history. Adjudication and summary events log their complete prompts. The full chat input is reconstructible (the system prompt is versioned in git, the findings are in the run, and history replays from prior audit events), so the trail is auditable, but README's "everything the model does touch is written verbatim into audit.jsonl" and the app footer's equivalent overstate what the chat events capture.

### F10 · Low · The UI renders an em dash as the null-NDC placeholder

`FindingsTable.tsx:88` renders `"—"` when a finding has no NDC, and the live app displays it (verified on the Cyanocobalamin row of the live sweep). The memo shares the pattern (see F1). A placeholder glyph is arguably typography rather than prose, but the register rule as written says none, and "n/a" (which `findings.md` already uses at `exports.py:115`) would satisfy it.

### F11 · Info · Two narrative numbers, one exact and one loose

The journal's over-matching story reproduces exactly: run `20260709T222103Z` recorded 530 findings and 199 candidates, and the next run recorded 55 findings and 40 candidates. The same journal entry says the fixes cut candidates to "~30 real ones"; the logs say 40. The 6x cost-cut estimate derived from it is in the right range (199 to 40 candidates is 20 batches down to 4). Exact where it matters, loose in one place worth a one-word correction.

### F12 · Info · The style guide file has no charset declaration

`design/styleguide/index.html` starts at `<title>` with no `<meta charset>`, so any server that omits a charset header renders its em dash and middot glyphs as mojibake (observed this session via a plain `http.server`). Opening the file directly is unaffected. One line fixes it if the guide will ever be served.

### F13 · Info · The synthetic sample formulary collides with a real FDA NDC

Sample item "Ketorolac Tromethamine" carries synthetic NDC 0409-3613-01, which exact-matches a real package NDC inside FDA's Bupivacaine shortage group. Demo users therefore see an `exact_ndc` shortage finding pairing a ketorolac line with a bupivacaine record (finding 13 of the live run). The drawer discloses the mismatch, and it is honestly an illustration of the tool's own "exact NDC is only as truthful as your formulary's NDC" edge, but anyone using the demo to evaluate match quality may read it as a bug. Worth either a different synthetic NDC or a README sentence.
